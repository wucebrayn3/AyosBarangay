THIS IS HOW THE BACKEND SHOULD BE TESTED WITH POSTMAN OR frontend

1) Users
api endpoint
GET /api/users/
POST /api/users/
GET /api/users/{id}/
PATCH /api/users/{id}/
DELETE /api/users/{id}/

headers (specify what type user can access)

GET: Public (no auth required by current permission logic)
POST/PATCH/DELETE: admin role only
Typical headers:
Content-Type: application/json
Authorization: Basic ... or session auth (for protected methods)
payload

{
  "username": "juan123",
  "first_name": "Juan",
  "last_name": "Dela Cruz",
  "email": "juan@example.com",
  "role": "resident",
  "purok": "Purok 1",
  "phone_number": "09171234567"
}
explanation (what is the endpoint for and why is it there)
Manages platform user accounts and roles for RBAC.

2) Puroks
api endpoint
GET /api/puroks/

Creates and manages purok zones used for organizing and tracking concerns per area.

Real-time (WebSocket) for Testing
A) All issue events
api endpoint
WS /ws/issues/

headers (specify what type user can access)
Currently open unless restricted by deployment middleware; can be used by clients needing live updates.

payload
Client sends none required after connect.
Server pushes event payloads like:

{
  "type": "event",
  "entity": "infrastructure_issue",
  "action": "updated",
  "data": {
    "id": 12,
    "status": "in_progress",
    "purok_id": 3
  }
}
explanation
Streams live updates so frontend users do not need constant polling.

B) Purok-specific events
api endpoint
WS /ws/issues/purok/{purok_id}/

headers (specify what type user can access)
Same as above.

payload
Same event format, scoped to specific purok group.

explanation
Lets purok-focused dashboards receive only local area updates.
